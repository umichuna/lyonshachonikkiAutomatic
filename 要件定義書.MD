# 社長日記 自動生成・公開システム 要件定義書 兼 設計書

作成日: 2026年7月10日

---

## 1. 背景・目的

現在、社長日記(Vol.001〜)のHTML記事は、Claudeとのチャットで都度デザイン説明をしながら手動作成し、EdgeOne Pages Dropでホスティング、Google Sitesに埋め込む、という流れで運用している。

この運用を、**「タイトル・本文・写真を渡すだけで、公開URLの発行とスプレッドシート記録まで自動で完了する」**状態まで簡略化することが本システムの目的。

---

## 2. 対象範囲(スコープ)

含む:
- HTML記事の自動生成(既存デザインテンプレートへの流し込み)
- GitHub Pagesへの自動デプロイ・URL発行
- 発行URL・タイトル等のGoogle Sheetsへの自動記録

含まない(将来検討):
- Google Sites側への自動埋め込み(手動のまま)
- 記事本文そのものの自動執筆(文章は引き続き人が用意 or Claudeと相談して作成)

---

## 3. 全体フロー

```
① 記事の元ネタ準備
   (タイトル・本文・写真をClaudeプロジェクトで下書き作成 ※既存の運用)
        ↓
② HTML生成
   (固定デザインテンプレートに流し込み、vol00X.html を生成)
        ↓
③ GitHubリポジトリへpush
   (1リポジトリ内にHTMLファイルを追加)
        ↓
④ GitHub Actions起動
   ├─ GitHub Pagesへ自動デプロイ → 公開URL発行
   └─ 発行されたURL・タイトル・日付をGoogle Sheetsへ追記
        ↓
⑤ (手動)Google Sitesに埋め込み
```

---

## 4. 機能要件

| # | 機能 | 内容 |
|---|------|------|
| F-1 | HTML生成 | タイトル・本文・写真を受け取り、Vol.001〜004と同一デザイン(しっぽり明朝+クレーOne+Cormorant Garamond、cream/beige/brownパレット、Firebaseリアクションボタン5種)でHTMLを生成する |
| F-2 | 自動デプロイ | 生成したHTMLをGitHubリポジトリにpushし、GitHub Pagesとして公開する |
| F-3 | URL発行 | デプロイ後、`https://[ユーザー名].github.io/[リポジトリ名]/volXXX.html` 形式のURLが自動生成される |
| F-4 | スプレッドシート記録 | 発行されたURL・記事タイトル・公開日を指定のGoogle Sheetsに自動で1行追記する |
| F-5 | リアクション機能維持 | 既存のFirebase Firestore連携によるリアクションボタン(👍❤️😊💪📖)を新規記事にも自動で組み込む |

---

## 5. 非機能要件

| 項目 | 内容 |
|------|------|
| 運用頻度 | 月数回程度(不定期更新) |
| 実行トリガー | 手動起動(GitHub Actionsのworkflow_dispatch、もしくはリポジトリへのpushをトリガーに) |
| コスト | 既存のGitHub(無料枠)・Google Sheets APIの範囲内で完結。追加課金なしを想定 |
| デザイン一貫性 | 過去のVol.001〜004と見た目が完全に揃うこと(テンプレート化により担保) |
| 保守性 | リポジトリは1つに集約し、記事はファイル追加のみで増やせる構成とする |

---

## 6. システム構成・技術スタック

- **HTML生成**: Claude Code(テンプレートエンジン的にHTMLを組み立てるスクリプト、もしくはClaudeプロジェクトでの下書き→変換)
- **バージョン管理/ホスティング**: GitHub リポジトリ + GitHub Pages
- **自動化**: GitHub Actions(既存のパン製造数予測システムと同じ仕組みを流用)
- **データ記録**: Google Sheets API(既存のダッシュボード自動化と同じ認証・連携方式を流用)
- **リアクション機能**: Firebase Firestore(既存資産をそのまま利用)

## 8. 処理フロー詳細

1. Claudeプロジェクト(社長日記制作)でタイトル・本文・写真を渡し、下書きHTMLを作成
2. 完成したHTMLファイルをリポジトリの所定フォルダに配置しpush
3. push検知でGitHub Actionsが起動
   - GitHub Pagesへの反映(自動、数十秒〜数分で公開)
   - 発行URLを取得(リポジトリ名・ファイル名から組み立て、もしくはActions実行結果から取得)
   - Google Sheets APIを叩き、タイトル・URL・日付を1行追記
4. 通知(任意):Discordなどに「Vol.005を公開しました」と通知させることも可能(既存のパン予測システムと同じ仕組み)

---

## 9. 実装ステップ(案)

1. **Step1**: Vol.001〜004のデザインをテンプレート化(共通CSS/JS・写真埋め込みロジックの切り出し)
2. **Step2**: 新規記事HTMLを生成するスクリプトをClaude Codeで作成
3. **Step3**: GitHubリポジトリ作成、GitHub Pages有効化
4. **Step4**: GitHub Actionsワークフロー作成(デプロイ+Sheets書き込み)
5. **Step5**: Vol.005で試験運用し、URL発行・シート記録が問題なく動くか確認
6. **Step6(任意)**: Discord通知の追加

## 11. 入力アプリ 画面設計(タブ切り替え式・4画面)

### 画面1:基本情報・本文入力
- 入力項目:Vol番号(自動採番+手動修正可)、タイトル、サブタイトル、本文
- 強調・見出しの入力は「右クリックメニュー」方式(スマホは長押し)で行う
  - 文字を選択して右クリック→ [太字][下線][マーカー][引用(四角囲み)][見出しにする] から選択
  - 何も選択せず行にカーソルがある状態で右クリック→ [見出しにする] を選ぶとその行全体が見出しになる
  - 裏側では `**太字**` `__下線__` `==マーカー==` `>> 引用` `## 見出し` の記法に自動変換されるが、利用者が記号を意識する必要はない
- 画面内に常設の「入力方法」ボタンを設置し、押すと早見表をポップアップ表示(太字/下線/マーカー/引用/見出しそれぞれの操作方法を一覧化)。フォーマットのダウンロードは不要とする
- 文字数(ボリューム)は自動カウント表示、目安に対して自分で調整可能

### 画面2:写真設定
- 画面1の本文から見出しを自動抽出し、見出しごとに写真をひも付けて選択
- 例:「新しい門出」という見出しに写真Aを割当
- ルール:記事全体で写真は最大4枚、1見出しにつき1〜2枚まで
  - 超過時はエラー表示(「1見出しにつき2枚までです」等)
  - 4枚を超える写真が必要な場合は「城野へご依頼ください」の注意書きを表示
- アップロード可能形式:jpg / png のみ、1枚あたり5MB以下

### 画面3:プレビュー・掲載
- プレビュー用に仮ホスティングしたページを表示、または発行前URLを自動で開く
- メッセージボックス:「プレビューをご確認いただき、修正点がなければ『掲載する』を押してください」
- 「掲載する」クリック→確認ダイアログ「掲載処理を実行します。実行するとスタッフ全員に通知されます。よろしいですか?」
- OK後の処理:
  1. ボタンを無効化(連打・二重送信防止)
  2. 同一Vol番号がスプレッドシートに既存でないかチェック(あれば書き込み中止しエラー表示)
  3. スプレッドシートへ1行転記(この転記をトリガーに、既存の通知システムが自動で走る仕様のため本システム側での追加通知処理は不要)
  4. 転記成功後、その記事を「掲載済み」ステータスにして以後の編集を不可にする

### 共通仕様
- 画面1〜3はタブで切り替え、いつでも前の画面に戻って部分修正が可能
- 各画面の入力内容は自動保存され、途中で閉じても「前回の続きから再開しますか?」と復元可能
- 掲載履歴は転記先スプレッドシートに直接リンクし、アプリ内からもいつでも参照可能

---

## 12. 承認・運用ルール

- 利用者:最大3名(あなた含む社内担当者)を想定。外部共有はしない
- 承認方式:掲載時に操作している本人が「掲載する」を押すことで承認完了とする(第三者による二重チェックは設けない)
- 承認者の記録:掲載処理実行時のユーザー名をスプレッドシートに記録する

## 13. エラー・通知設計

- GitHub Pagesへのデプロイ失敗、スプレッドシートへの書き込み失敗が発生した場合、担当者(チナミさん)宛にDiscord通知を送信する
- スタッフ向けの通知(新着記事の周知)は、スプレッドシートへの転記をトリガーに既存システムが自動実行するため、本システムでは対応不要(実装済み)
- クライアント向けの不具合連絡は「不具合等ございましたらLINEにてご連絡ください」の注意書きを画面に表示するのみとし、システム上の自動連携は行わない
